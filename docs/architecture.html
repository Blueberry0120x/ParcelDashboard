ENGINE CONTRACT — WHO OWNS WHAT (BINDING RULES)
! The Toolbox Rule — One Tool Per Job, No Overlap

Why this contract exists: The system drifts because two engines (MapEngine and SetbackEngine) share coordinate math with no single owner. A "small change" in one engine's transform silently breaks the other. This contract assigns every operation to exactly one engine. If a change would cross a boundary, it must be discussed first — not just coded.

The Golden Rule: If you need to cut wood, you pick up the saw. You do NOT pick up the hammer and try to bend the wood into shape. Every operation below has exactly one engine that handles it. Use the wrong engine and the system breaks.

2 The Coordinate Transform Contract (Root Cause of Drift)

This is the exact spot where drift happens. Two coordinate frames exist in the system. Every bug in the last month traces back to mixing them up.

Frame A: Geographic (lat/lng)
Owner: MapEngine ONLY
Units: decimal degrees (WGS84)
Origin: state.lat, state.lng (the pin)
Used for: Leaflet polygons, markers, lat/lng display
Conversion factors: F_LAT = 364,566 ft/deg, F_LNG = 365,228 * cos(lat) ft/deg
Rule: NO other engine touches these values or does this conversion.
Frame B: Local Rotated (feet)
Owner: SetbackEngine (for all geometry math)
Units: feet from lot center
Origin: (0,0) = lot center (same point as state.lat/lng)
Axes: +X = toward rear, +Y = toward right side
Used for: building offsets, setback distances, chain dim positions, extents
Rule: ALL geometry math happens here. Results handed to MapEngine for drawing.

The handoff protocol (MUST follow):

  1. SetbackEngine computes geometry in Frame B (local feet). It returns positions, extents, and measurements in feet.
  2. MapEngine receives those feet values and converts to Frame A (lat/lng) using the rotation transform: rx = px*cos(rot) - py*sin(rot), ry = px*sin(rot) + py*cos(rot), then [lat + ry/F_LAT, lng + rx/F_LNG].
  3. MapEngine creates/updates Leaflet layers with the lat/lng values.
  4. NEVER skip step 2. SetbackEngine must NEVER convert to lat/lng itself. MapEngine must NEVER compute extents or distances in feet itself.

siteNorthDeg — stored separately from rotation. Controls only the compass SN arrow direction (reference bearing for drawings). Never affects lot geometry, building positions, or frame transforms. Draggable on the compass widget.

Polygon vertex frame (the KB-1 bug): When a parcelPolygon exists, the polygon vertices arrive as lat/lng (Frame A). They must be converted to Frame B (local feet relative to the pin) ONCE at the start of updateBldgDimLabels(), using the pin as origin — NOT the polygon centroid. The current code uses the centroid as origin, which is why KB-1 exists. The fix: convert polygon vertices to local feet using state.lat/state.lng as the origin, not ccLat/ccLng.

3 Change Protocol — Rules Before Modifying Any Engine

These rules exist because the system is stable when these boundaries are respected, and it drifts when they are crossed:

RULE 1: One engine, one responsibility. If a function already exists in the ownership table above, do NOT reimplement it in another engine. Call the existing function. Duplication is how drift starts.
RULE 2: Frame B math stays in SetbackEngine. Any calculation involving feet, distances, perpendicular measurements, extents, or building bounding boxes is SetbackEngine territory. MapEngine does not compute distances — it draws.
RULE 3: Frame A conversion stays in MapEngine. Any conversion from feet to lat/lng (or vice versa) is MapEngine territory. SetbackEngine does not know what latitude is.
RULE 4: No "small fixes" that cross boundaries. If a fix requires both Frame A and Frame B work, it is NOT a small fix. It crosses the boundary. State the boundary crossing explicitly before writing code.
RULE 5: Test on three lots. Any change to coordinate math or dimension drawing MUST be visually verified on: (1) a simple rectangle lot, (2) a rotated rectangle lot, and (3) a polygon lot. If it works on one but breaks another, the fix is wrong.
RULE 6: ConfigEngine is passive. ConfigEngine never calls other engines. It holds data. Other engines read from it and write to it. If ConfigEngine starts having import dependencies on MapEngine or SetbackEngine, the architecture is broken.
RULE 7: ExportEngine.save() is the ONLY write path. No engine may write to localStorage, POST to the server, or trigger a file download except through ExportEngine. This is non-negotiable.

FLOW A: FULL ADDRESS INTAKE PROCEDURE  ·  build.py  data setup — no browser engine runs during this step
A1 Starting Point — The Address Is All You Have

When a designer hands you a street address for a new site, they are typically giving you a single line: a city, a street number, and a state. Something like "2921 El Cajon Blvd, San Diego CA 92104". There are no attachments. No parcel number. No zoning printout. No lot dimensions. Just that string. This is normal. This is how every site in the system started.

Your job — before the tool can render a single building footprint or calculate a single setback — is to transform that address into a JSON file with over forty verified fields. Each field has an authoritative source. None are guessed, assumed, or derived from memory. When the data flows from your research into the JSON, and from the JSON through the build pipeline into the browser, the feasibility numbers the designer sees must be traceable back to a primary source: an assessor record, a municipal code section, a city fee schedule PDF. That traceability is the entire point of this intake process.

The intake process has a fixed order. You work through nine steps in sequence because some pieces of data are required to unlock the next step. You need the APN before you can look up the polygon geometry. You need the zoning designation before you can download the right code section. You need the code section before you can fill the setback, FAR, and density fields. Follow the order not as bureaucratic procedure, but because the data has natural dependencies — each step builds on the last.

As you research, document everything in real time. For any field you find, write the exact value. For any field you cannot yet confirm, write the TBD string: "-- TBD: [what it is] | Source: [where to get it]". This format matters. A TBD string is visually distinct in the browser UI. When the designer looks at the zoning table and sees -- TBD: check CCHS tier map, they know at a glance that this number has not been confirmed and they should not draw revenue conclusions from it. A zero, by contrast, looks like real data and will silently feed into every downstream calculation.

The site JSON has two top-level keys: .site and .saved. Understanding the difference matters from the start. The .site key holds the permanent public record — the facts from the assessor, the zoning code, the fee schedule. These are what the inspector contacts tab shows, what the zoning table renders, what the feasibility calculator uses. The .saved key holds live session state — which building footprints are currently placed, what rotation the lot is at, what setbacks are currently active on the map. The build pipeline reads both. But the live save endpoint — the one that fires every time the user clicks Save — only ever overwrites .saved. The .site data is never touched at runtime. It changes only when you manually edit the JSON file and rebuild. This separation protects your research from being overwritten by a user's session activity.

A2 Step 1 — Parcel Identity Lookup

What you are trying to find: the Assessor's Parcel Number (APN), the legally recorded lot area in square feet, the lot dimensions if available, the legal description, and the year built. These are the ground-floor facts — every other piece of research in this intake process builds on top of them.

Where to start: Open ParcelQuest at assr.parcelquest.com. This is the primary parcel database for California counties and gives you clean assessor data with minimal navigation overhead. It is also linked in the app's suite bar. When the site loads, a search bar appears at the top and a map fills the center of the screen. Type the address the designer gave you into the search bar — include the full address, city, and state abbreviation — and press Enter. The system attempts to match the address string against the county's parcel records.

When a result appears, examine the map carefully. The system places a pin on what it believes is the correct parcel. Before recording anything, cross-reference that pin visually with the satellite view: does the pin land on the correct building, on the correct block face, on the correct side of the street? In dense urban neighborhoods, an ambiguous or slightly off address can silently match the wrong lot — a neighboring parcel, the back unit of a duplex, or a separately recorded commercial stub. Confirm the parcel address shown in the detail panel exactly matches what the designer gave you. If there are multiple search results, a list of matches will appear; read each one's address and select only the one where the address, city, and ZIP all match precisely.

With the correct parcel confirmed, read the record detail panel. The first item of critical importance is the Assessor's Parcel Number, or APN. In San Diego County, this is a ten-digit number formatted with hyphens, looking like 469-690-20-00. In King County, Washington, it is a ten-digit number with no hyphens: 0723049368. The format varies by county but the function is identical in every jurisdiction: the APN is the unique legal identifier for this exact piece of land in the county's official records system, and it is the key used by every other government database — the zoning database, the permit tracking system, the GIS parcel layer — when looking up information about this property. Write it down exactly as displayed, including hyphens. Do not strip or reformat it.

Next, record the lot area in square feet. ParcelQuest shows this in the parcel detail under a label like "Lot Size," "Land Area," or "Parcel Area." This is the authoritative number for the lotSF field. Always use the assessor's published lot area figure — never calculate it yourself from lot width times lot depth. The reason is straightforward: most urban lots in older cities are not perfect rectangles. Their boundaries were drawn from legal descriptions, old plats, and deed calls that create slight irregularities at the corners. The assessor's recorded lot area accounts for the true survey polygon including those irregularities. If you compute width × depth, you will get a number that is close but not exactly correct, and that small discrepancy compounds into systematic errors in the density calculation (floor(lotSF / densityPerSF)) and the FAR ceiling calculation (baseFAR × lotSF = max buildable SF).

If the assessor record shows lot width and depth — not all records do — record those as well. They go into lotWidth and lotDepth. These two dimensions drive the rectangle placeholder that the map engine draws when no parcelPolygon array is provided. For a genuinely rectangular lot they are accurate; for an irregular lot they are approximate and the polygon will visually override them once it is entered.

Record the legal description if shown. This is the deed reference — typically a tract name, lot number, and block number from the original recorded subdivision map, for example: LOT 5 BLK 3 SUBDIVISION MAP 2105. The legal description matters less for the tool's direct function but is useful when ordering a title report, verifying ownership in county records, or when a city agency asks for documentation confirming the parcel's legal identity. Enter it as a string in the legalDescription field. If ParcelQuest does not display it, use the TBD format referencing the county recorder as the source.

Record yearBuilt if the record shows existing structures. This is informational for the designer — it helps them understand the context of the lot (whether existing buildings need to be demolished, whether there are potential historic preservation flags for structures over a certain age). Use a four-digit year string. If the parcel is currently vacant land, use an empty string "".

When ParcelQuest does not have data for the jurisdiction: ParcelQuest covers most California counties well but has limited or no coverage of Washington state and some smaller California jurisdictions. For King County, Washington parcels, go to the King County Assessor at blue.kingcounty.com/assessor/eRealProperty. The search interface accepts the full address or the parcel identification number. The result shows the parcel detail including the property account number (equivalent to APN), lot area, legal description, and year built. The county GIS viewer at gismaps.kingcounty.gov also provides parcel information through an interactive map search. For any jurisdiction where the primary portal is unclear, Regrid at app.regrid.com is a reliable national fallback. Regrid aggregates county parcel data from assessor sources across the United States and often surfaces data that proprietary portals present poorly or incompletely. Search by address, click the parcel on the map, and read the detail panel on the right side of the screen.

address string Full street address with city, state, and ZIP. Use exactly what the assessor record displays — not what a mapping service autocompletes.
apn string Assessor Parcel Number. Include hyphens exactly as the county records them. Do not strip formatting — some downstream systems query by formatted APN.
legalDescription string Tract/lot/block from deed. From ParcelQuest "Legal" tab or county recorder. Use TBD with recorder URL if not shown in ParcelQuest.
yearBuilt string Four-digit year of existing structure(s). From assessor detail panel. Use "" if vacant land — not "0", which implies a year.
lotSF number Total lot area in square feet. Always use the assessor's recorded figure. Never compute from W × D — irregular parcels will not match.
lotWidth number Front-face width in feet. From assessor or original tract map. Use 0 if not available; parcelPolygon will override visually.
lotDepth number Front-to-rear depth in feet. From assessor or original tract map. Use 0 if not available.
A3 Step 2 — Parcel Coordinates and Boundary Geometry

What you are trying to find: the precise latitude and longitude of the parcel's geographic center, and — for any lot that is not a simple rectangle — the complete ordered list of GPS coordinates for every corner of the lot boundary.

Getting the centroid latitude and longitude: Open Google Maps in a browser tab. Type the property address into the search bar and press Enter. The map will zoom to the address and place a familiar red location pin. Now — before right-clicking — mentally locate the visual center of the lot itself, not the center of the building that sits on it. The standard address pin often lands on the front door or the building's street-facing wall, which is at the front of the lot, not at its geographic midpoint. For the tool's map to center correctly on the parcel, you want the approximate geographic center of the land area. Look at the satellite view to see where the lot boundaries are, then identify the midpoint of the lot.

Right-click on that spot — the approximate center of the lot — directly on the map. A small context menu appears. At the very top of this menu, on a gray background, you will see a pair of decimal numbers such as 32.7556, -117.0919. These are the coordinates in decimal degree format. Click this pair of numbers: Google Maps automatically copies them to your clipboard. The first number is the latitude (the north-south position, a positive value for any location in the continental United States, ranging roughly from 24 to 49 degrees). The second number is the longitude (the east-west position, a negative value for all continental US locations because they are west of the Prime Meridian, ranging roughly from -66 to -125 degrees). If you ever see a positive longitude number, something has gone wrong with the copy — discard it and try again.

Paste the two numbers into the JSON as saved.lat (the positive number) and saved.lng (the negative number). Use at least six decimal places — this gives you positional accuracy to within about 11 centimeters on the ground, which is far more than sufficient for this tool's purpose. The centroid is where the Leaflet map will center when the site first loads in the browser, and it serves as the reference origin for the proj4 coordinate conversion when generating a CAD LISP export.

Deciding whether you need polygon geometry: Every site uses one of two approaches for drawing the lot boundary — rectangle mode or polygon mode. Rectangle mode uses the lot width and depth (from the assessor record) plus a rotation angle to draw a simple four-cornered rectangle. This is the correct approach for standard residential lots that were created by a gridded subdivision plat. Polygon mode uses a full array of vertex coordinates to draw the actual lot shape regardless of complexity. You need polygon mode when the lot is non-rectangular: L-shaped, flag-shaped, pie-shaped, wider at the rear than the front, or created from a combination of multiple APN records. You also need polygon mode when the fronting street runs at an angle that makes the lot appear skewed relative to the map grid.

To decide which mode to use, look at the lot in satellite view. If the lot appears to be a clean, four-sided rectangle with right angles at all corners, rectangle mode is sufficient — leave parcelPolygon as an empty array and the tool will construct the rectangle from lotWidth, lotDepth, and the rotation you set during the alignment step. If the lot has any more complex shape, extract the polygon.

Extracting polygon vertices from the GIS portal: Open the county GIS parcel viewer for the jurisdiction. For San Diego County parcels, use the SanDag GIS portal at arcgis.com under the San Diego County open data portal, or the County of San Diego's interactive parcel map. Search by APN. When the parcel is selected, the GIS viewer shows the parcel boundary as a highlighted polygon. You need the coordinates of each vertex of that polygon in order. Some GIS portals expose the vertex coordinates in the feature detail panel — look for tabs labeled "Geometry," "Coordinates," or "Attributes." Others require you to download the parcel as a GeoJSON or KML file, which you can then open in a text editor to read the coordinate array.

For King County, Washington, use the King County GIS parcel viewer at gismaps.kingcounty.gov. For Orange County, use the OC Public Works GIS parcel viewer. For any jurisdiction where the GIS portal is unclear, Regrid (Step 1 fallback) sometimes provides a polygon export option from the parcel detail panel.

Once you have the polygon data, format it as an ordered array of [latitude, longitude] pairs, one per vertex, starting from the northwest corner and proceeding clockwise around the perimeter. Each consecutive pair of coordinates defines one edge of the lot. The last vertex connects back to the first one, implicitly closing the ring. For example, a four-sided rectangular lot in San Diego might look like: [[32.756, -117.092], [32.756, -117.090], [32.754, -117.090], [32.754, -117.092]]. Store this array in the parcelPolygon field. Also save the raw GeoJSON source file to reference/StateRegulation/{state}/{city}/ as a permanent backup that can reproduce the polygon if needed.

Selecting the CAD coordinate zone: The cadZone field tells the tool which State Plane Coordinate System projection to use when converting decimal-degree GPS coordinates into surveying feet for the AutoCAD LISP export. The United States is divided into approximately 120 State Plane zones by NOAA; each zone is optimized for minimal distortion within a bounded geographic region. Pick the zone based on the parcel's location: CA_VI covers all of San Diego and Imperial County; CA_V covers Los Angeles, Orange County, and Ventura County; CA_IV covers Fresno, Santa Barbara, and San Luis Obispo; CA_III covers the San Francisco Bay Area, Sacramento Valley, and the Central Coast north of Santa Barbara; WA_N covers Seattle, Bellevue, Redmond, Everett, and the entire greater Puget Sound area north of and including Olympia; WA_S covers Thurston, Pierce, Lewis, Grays Harbor, and Pacific counties. If you are uncertain which zone applies to a specific city, search for the projected coordinate system at epsg.io using the city name.

saved.lat number Centroid latitude in decimal degrees. Six decimal places minimum. From Google Maps right-click → first number in the coordinate pair.
saved.lng number Centroid longitude in decimal degrees. Always negative for continental US. From Google Maps right-click → second number in the coordinate pair.
cadZone string State Plane projection zone code. Determined by geographic location. Must match a proj4 definition registered in ConfigEngine.
parcelPolygon array [[lat,lng], ...] ordered clockwise from NW corner. Required only for non-rectangular lots. Leave [] for clean rectangular lots.
A4 Step 3 — Zoning Code Identification and Development Standards

What you are trying to find: the official zoning designation code for the parcel, and then every specific development standard from the municipal code that applies to that zone — the setbacks, height limit, FAR, density, and commercial depth rules that together define the maximum envelope of what can be built on the lot.

Finding the zone designation: The fastest method is the city's own zoning inquiry tool. For San Diego, this is the ZIMS (Zoning Inquiry Management System), accessible from the city's Development Services portal at sandiego.gov. Enter the street address or APN and the system returns the base zone designation and lists any overlay zones that appear in the city's records. The zone code you receive will look something like CC-3-9 (Community Commercial, 3-story tier, 9 DU/acre), RM-3-7 (Residential Multiple, 3 stories, 7 DU/acre), or CUPD-CU-2-4 (Commercial Urban Planned Development). For Garden Grove, navigate to the city's Community Development GIS portal and toggle on the Zoning layer, then click the parcel to see its designation. For Burien, use the city's online parcel viewer or go directly to the municipal code at codepublishing.com/WA/Burien where the zoning tables are organized by zone code. For Escondido, the Planning Division administers zoning and can confirm the designation by address over the phone or through an online form on their planning portal.

Record the zone designation exactly as the city presents it, including all hyphens, letters, and numbers in the precise sequence shown. The designation is not just a label — it is the lookup key into the municipal code's regulation tables. R-3 and R-3-A are different zones with different development standards. CUPD-CU-2-4 is not the same as CU-2-4. Never abbreviate, interpret, or simplify the code string. Write it verbatim.

Finding the municipal code and navigating to the right section: Navigate to the city's online municipal code. Most California and Washington cities publish their code through Municode at library.municode.com or through American Legal at amlegal.com. Search for the city by name on either platform. Once inside the city's code, you need to navigate to the zoning chapter. In San Diego this is Title 13 (Land Development Code), then the relevant article by zone type: Article 1 for general regulations, Article 2 for residential zones (RM, RT, RS zones), Academic 3 for commercial and mixed-use zones (CC, CN, CV, CR zones). In other cities the structure varies but the zoning chapter is almost always titled "Zoning," "Land Use," or "Development Code" and appears in the middle sections of the municipal code.

Within the zoning chapter, navigate to the specific zone you found. Each zone has a dedicated code section that contains a development standards table — typically a two-column table listing each requirement (minimum lot area, setbacks, maximum height, FAR, minimum parking) alongside the numerical standard for that zone. Some cities use a consolidated table listing all zones side-by-side; others have individual sections per zone. Either way, your goal is to find the row or column that corresponds to your zone designation and read every dimensional constraint it specifies.

Extracting the setback values: The table lists minimum required setback distances — measured in feet from the relevant property line to the nearest wall or structural element of any new building. Record the front setback (from the front property line, which typically faces the street), the rear setback (from the rear property line, at the far end of the lot from the street), and the side setback (from the interior side yard property lines, the lines that separate the lot from its neighbors). Some zones specify different setback requirements for the two side yards, or distinguish between an interior side yard and a street-side yard (when the lot is a corner lot with street frontage on two sides). Record whatever the code specifies. If only one side setback is listed, use that value for both sideL and sideR when seeding .saved.setbacks.

Extracting the FAR and height values: The Floor Area Ratio is a multiplier. A FAR of 1.5 on a 7,500 SF lot means you may build at most 11,250 SF of total floor area across all floors combined. Enter this as baseFAR. For purely residential zones in many California cities, there is no explicit FAR limit — density is controlled entirely by unit count and lot coverage percentage instead. In those cases, set baseFAR to 0. The commFAR field holds the FAR applicable to the commercial portion in a mixed-use zone, or it may hold a higher bonus FAR available through a density bonus program such as California's Complete Communities Housing Solutions (CCHS). Set it to 0 if neither applies.

For height, record both the unadjusted base height limit and the maximum allowed height (which may be higher if bonus programs apply). The baseHeightLimit field holds the standard code-specified height with no bonus considerations. The maxHeight field holds the absolute ceiling — the highest any structure on this lot could legally reach given all available bonus programs. For San Diego sites, the cchsMaxHeight field holds the Complete Communities Housing Solutions maximum height for the applicable CCHS tier; for sites outside San Diego or in areas where CCHS does not apply, set this equal to maxHeight.

Extracting density: In California residential zones, density is most commonly expressed as a minimum required lot area per dwelling unit. The code reads something like "minimum lot area per DU: 600 SF" or "1 unit per 2,500 SF of lot area." Whatever form it takes, convert it to the JSON's densityPerSF field, which holds the number of square feet of lot area required per one dwelling unit. For "1 DU per 600 SF," densityPerSF = 600. For "maximum 25 DU per acre," first convert: 1 acre = 43,560 SF, so 25 DU per 43,560 SF means 43,560 / 25 = 1,742 SF per unit, so densityPerSF = 1742. The feasibility calculator computes the base unit count as floor(lotSF / densityPerSF). Set to 0 if the zone has no density rule expressed in SF per unit.

Handling conditional rules: Zoning codes frequently write rules with conditions: "maximum height is 30 feet, except that within 300 feet of a transit stop the maximum is 65 feet." Never guess which condition applies to this specific parcel without checking. Record the conservative value (the lower height) for now, and add a TBD note in the notes field: | TBD - C: maxHeight conditional — verify transit distance for 30/65 ft rule (SDMC §131.0430). The conditional check may require a GIS measurement, a phone call to the planning counter, or a specific map overlay that you will handle in Step 4 (A5).

A5 Step 4 — Planning Overlays and Special Districts

What you are trying to find: every overlay zone, special district designation, and planning-program tier that applies to this specific parcel, layered on top of the base zoning. Overlays are the hidden variables in land development feasibility. They can eliminate the parking requirement and save $30,000–$50,000 per stall in construction cost. They can unlock an extra 30 feet of height through a state housing program, adding entire additional residential floors. They can trigger a Coastal Development Permit that extends the approval timeline by 12–18 months. Or they can impose a height restriction below what the base zoning allows if the lot sits within an airport influence zone. The base zoning code you found in Step 3 tells you the standard rules. The overlays tell you which standard rules bend, which get overridden, and which additional requirements are added. You need to know all of them before any concept design work begins.

Unlike the base zone — which is one definitive answer from one system — overlays require checking each category independently because they come from different databases, maintained by different agencies, and governed by different regulatory frameworks. Some are city-administered (CCHS tier, mobility zone). Some are state-administered (TPA, Coastal Commission zone). Some are federal or regional (airport influence area under the Airport Land Use Commission). Work through each one systematically.

For the CCHS tier check in San Diego: the Complete Communities Housing Solutions program assigns parcels to one of three tiers based on proximity to transit infrastructure and policy goals. Navigate to sandiego.gov/planning/programs/housing and open the Complete Communities interactive map. Zoom to the parcel's address. The map layer shows color bands for Tier 1, Tier 2, and Tier 3. Read which tier the parcel falls in. Tier 1 allows modest height bonuses above the base code limit for projects with affordable units. Tier 2 allows medium bonuses. Tier 3 — typically parcels along high-frequency transit corridors and in the most urbanized neighborhoods — unlocks the highest bonus heights and allows up to 6.5 FAR for projects that meet affordability thresholds. Record the tier as a string in the planningAreas[] array. Also record the community plan area name here (e.g., "Mid-City: City Heights") because the community plan area name determines which fee schedule rows apply in Step 5.

For the Transit Priority Area (TPA) check in California: a TPA is officially defined as a half-mile radius circle centered on a major transit stop that has fixed-route bus service at 15-minute or better headways during peak commute hours, or any stop on a light rail, commuter rail, or ferry line. At SANDAG's GIS portal for San Diego, open or download the TPA boundary layer and confirm whether the parcel centroid falls within it. If yes, this is a TPA parcel. Being inside a TPA has significant feasibility implications: under San Diego Municipal Code §142.0528, residential uses are exempt from minimum parking requirements when located within a TPA. For Washington state parcels, check the Sound Transit and Metro Transit service maps to identify qualifying stops and buffer zones under Washington's Growth Management Act transit-oriented development provisions.

The distinction between false and null is meaningful for quality control: when you audit a partial site's outstanding TBDs, every null overlay entry appears as an item still requiring a verification check, while false entries are treated as confirmed absent and do not appear in the TBD audit.

A6 Step 5 — Fee Schedule and Development Costs

What you are trying to find: the specific financial charges the city imposes on new construction — the per-square-foot enhancement fees, the per-unit impact fees, and the affordability fraction required to qualify for density bonuses. These are not optional details to fill in later. They are direct numeric inputs to the financial feasibility calculation that the pre-application checklist computes, and they often make or break a project's feasibility. It is surprisingly common for a project that looks financially viable under base zoning to become infeasible once $35,000 per unit in DIF charges are correctly applied. These numbers belong in the JSON before the designer has any serious conversation about unit counts.

Every city that charges development impact fees publishes an annual fee schedule document, typically a PDF, updated each fiscal year. This document is almost always available on the city's Development Services portal, Permit Center page, or Finance Department website. Searching for "city name development services fee schedule 2026 PDF" will usually find it within the first few results. Download the current year's version and save it to reference/StateRegulation/{state}/{city}/ so you have the primary source accessible for the next session when a fee number needs to be verified.

The NEF (Neighborhood Enhancement Fee) is a San Diego-specific charge that applies in designated community plan areas. It is levied per square foot of new construction gross floor area — not per unit, but per square foot of total building area. The rate is not uniform across the city; it varies by community plan area. The fee schedule PDF contains a table organized by community plan area name. Find the row corresponding to the community plan area you recorded in Step 4 (A5). The rate appears as a dollar-per-square-foot figure, such as $0.41/SF. Enter this value as a plain decimal number in nefRatePerSF (e.g., 0.41). The pre-application checklist tool calculates total NEF as buildable SF × nefRatePerSF and displays the result in the fees section. For any Washington state parcel or for any California city that does not use a neighborhood enhancement fee, set nefRatePerSF to 0.

The DIF (Development Impact Fee) is charged per residential dwelling unit rather than per square foot. It represents the city's accounting of the proportionate infrastructure cost — new roads, parks, fire stations, library expansions, utility capacity improvements — attributable to the additional population generated by the new project. DIF rates in San Diego are also organized by community plan area. Open the same fee schedule PDF and find the DIF table. Locate the row for your community plan area and read the per-unit charge. Enter this as difPerUnit in dollars (e.g., if the fee is $21,400 per unit, enter 21400). The checklist engine multiplies this by the number of qualifying units to produce the total DIF line item.

The DIF waiver threshold (difWaiverSF) reflects a policy used by several cities and state programs to encourage the construction of smaller, more affordable units. If a dwelling unit's floor area is at or below a specified size — measured in square feet — that unit is either partially or fully exempt from the DIF. The threshold is stated in the fee schedule or in the city's development impact fee ordinance. For example, the City of San Diego exempts units at or below 500 SF from the residential DIF under certain programs. Enter the threshold size in square feet as a plain number; enter 0 if the jurisdiction has no waiver policy.

The affordability percentage (affordabilityPct) is the fraction of total units in a project — expressed as a decimal between 0 and 1 — that must be designated as income-restricted affordable housing in order for the project to qualify for a state density bonus. Under California Government Code §65915, the minimum affordability set-aside ranges from 5% at the lowest bonus tier to about 24% for the maximum bonus. The specific percentage required also depends on the income level of the restricted units (very low income, low income, or moderate income — each tier has different thresholds and bonus percentages). For this JSON field, enter the baseline required percentage that applies to the most common scenario for this project type. For Washington state, the equivalent requirement comes from the State Environmental Policy Act (SEPA) exemption thresholds and local inclusionary housing ordinances, which vary by city. Record as a decimal: 10% = 0.10, 15% = 0.15. If no affordability requirement applies, enter 0.

A7 Step 6 — Inspector and Agency Contacts

What you are trying to find: the names and direct contact information for the city staff who will review and inspect the project at each technical discipline — building, structural, electrical, mechanical, and grading. The designer references these contacts when they need to call the plan checker with a specific code interpretation question before submitting drawings, or when they want to request a pre-application meeting. Having the right person's name and direct extension saves the designer from navigating an automated phone tree to find a human who knows the project.

The most reliable approach is a direct phone call to the city's Permit Center counter. For the City of San Diego, the main Development Services line is (619) 446-5000. For other cities, the Permit Center phone number typically appears on the first page of the fee schedule PDF you downloaded in Step 5, or on the city's Development Services landing page in large text under "Contact Us." When someone answers, introduce yourself briefly — "I'm doing pre-application research for a proposed residential project at [address, APN]" — and ask who the assigned building inspector and plan check engineers are for that address. Many cities assign inspectors geographically by district, meaning the same individual handles all permits and inspections for an entire neighborhood or zone. Some cities assign plan reviewers by project type (residential vs. commercial) and then route to a specific desk based on the submission address. The person who answers the phone will know the routing rules and can give you the assigned reviewer's name and number.

Record each contact in the inspectors[] array. Each entry is an object with two keys: name holds the discipline label (such as "Building Inspector", "Structural Plan Review", or "Grading/Civil Engineer"), and val holds the person's full name and their direct phone number (such as "Maria Garcia (619-446-5123)"). The array should always have at least five entries, one per discipline, populated for each site regardless of whether you have the names yet.

If you cannot reach a real person on the first call, or if the Permit Center tells you assignments rotate and the specific inspector isn't known until the permit is submitted, use the TBD format for the val field: "-- TBD: call SD Permit Center (619) 446-5000". This tells the designer exactly who to call to resolve the TBD, rather than leaving a blank field that gives them no direction. Write the TBD string with the counter phone number so they can resolve it in a single call.

The five standard discipline rows to include for every site are: Combination Inspector — the multi-discipline building inspector who performs rough framing inspections, insulation inspections, and final occupancy sign-off; Structural — the plan check engineer who reviews structural drawings, lateral bracing, shear wall design, and foundation plans; Electrical — the reviewer for service entrance, panel sizing, branch circuits, and EV charging infrastructure; Mechanical — covers HVAC systems, plumbing, gas lines, and ventilation calculations; Grading/Civil — responsible for the site grading plan, stormwater management, drainage design, utility connections, and erosion control measures during construction. For smaller cities where one inspector or reviewer handles multiple disciplines, record that person's name in every applicable row and note in parentheses which additional disciplines they cover.

A8 Step 7 — Assemble the Skeleton JSON File

What you are doing: taking all the research you gathered in Steps 1 through 6 and entering it into a structured JSON file that the build pipeline reads, compiles, and injects into the browser. This is the moment all your field research becomes machine-readable data that the tool can display, calculate with, and save.

Always start from the template: Open reference/site-template.json in VS Code. This is the canonical schema — it defines every field the system reads, using the exact key names and exactly the nested structure the JS engines expect. Every field is pre-initialized to a neutral placeholder value: empty strings "" for text fields, 0 for numbers, false for booleans, null for conditional flags, and [] for arrays. Copy this file — do not rename it in place, create a copy — to data/sites/ with the new filename. The naming convention is {state-abbrev}-{street-number}_{StreetName}.json, all lowercase except the street name which is title-cased, with no spaces in the filename. A property at 2921 El Cajon Blvd in California becomes ca-2921_ElCajon.json.

Never copy from an existing site's JSON. If you copy from ca-4335_Euclid.json to start a new site file, you inherit Euclid's actual lat/lng, setbacks, buildings array, rotation angle, and potentially buildings already placed on the map. You will almost certainly miss one of these fields during cleanup and the new site will open with a neighbor's data silently embedded. The template is deliberately neutral. Always use the template.

Set the siteId field first: Open the new file in VS Code. Find the siteId field — it's near the top of the JSON. Set it to the uppercase version of the filename base: for ca-2921_ElCajon.json, the siteId is "CA-2921_ELCAJON". The build system uses this string as the lookup key when compiling __SITE_LIST__. When a user selects this site from the dropdown, the browser sends this exact siteId string to the server as part of the activate request. When site-data.json is updated to point to this site, the activeSiteId value must be this exact string — character for character. A mismatch between the filename convention and the siteId will cause the site-switcher to malfunction or the site to fail to activate.

Filling the .site block with confirmed data: Work through the .site object field by field, from top to bottom. For every field where you have a confirmed, source-verified value from your Steps 1–6 research, enter the value directly. For every field where you do not yet have verified data — whether it requires a phone call you haven't made, a PDF you haven't downloaded yet, or a GIS check you haven't done — write the TBD string: "-- TBD: [description] | Source: [specific location]". The two-part format is important: the first part names what piece of information is missing, and the second part names exactly where to find it. "TBD" alone is useless. "-- TBD: DIF per unit for Mid-City community plan area | Source: City of San Diego fee schedule 2026, Table 4B" tells the next person or the next session exactly what to look for and where. The more specific the source instruction, the faster the TBD gets resolved.

Seeding the .saved block: The .saved key holds live session state. At skeleton stage, fill only the values you can confidently set right now. Set lat to the centroid latitude you found in Step 2. Set lng to the centroid longitude. Set rotation to 0 — you have not yet opened the map and visually aligned the lot, so 0 is the correct placeholder that means "not yet set." Copy the setback values from .site into .saved.setbacks as an object {front, rear, sideL, sideR} — this is intentional duplication. The map engine reads its active working setbacks from .saved.setbacks at runtime, not from .site. If you don't copy them here, the map will open with zero setbacks. Set buildings to an empty array [] and locked to false. Leave all other .saved fields at their template default values.

Writing the notes field: After entering all the data you have and flagging the unknowns, write a structured summary in the notes field. This is your work order for the next session. Format each outstanding item as a line starting with a pipe character: | TBD - {section}: {field} -- {source}. The section letter corresponds to these eight categories — A: Identity, B: Lot, C: Zoning, D: Overlays, E: Fees, F: Project, G: Inspectors, H: Coordinates. For example: | TBD - C: cchsMaxHeight -- check CCHS tier map at sandiego.gov/planning and | TBD - E: nefRatePerSF -- City Heights row, SD fee schedule 2026 and | TBD - G: all 5 inspectors -- call SD Permit Center (619) 446-5000. When you open this site in a future session, the notes field is the first thing you read. It is your complete, prioritized work order with no hunting required.

This file is the master registry that the build system reads to compile window.__SITE_LIST__, which populates the site-switcher dropdown in the browser. Add a new entry to the sites array with these fields: id (the siteId string you set), label (a short 3–6 word human name for the dropdown display, e.g., "2921 El Cajon"), address (full address string), file (the filename relative to the data/sites/ folder, e.g., "ca-2921_ElCajon.json"), state (two-letter abbreviation), city (city name), zoning (zone designation string), status (set to "skeleton"), and lastUpdated (today's date as a string). Save the file. The next build will pick up this new entry and add it to the dropdown.

A9 Step 8 — Build, Activate, and Align the Map

What you are doing: running the build pipeline to compile the new site into the browser-ready HTML, activating that site in the dev server, and then performing the first visual alignment — dragging and rotating the lot overlay until it lines up precisely with the real parcel visible in the satellite imagery. This is the moment the site goes from a JSON file on disk to a live interactive map you can work with.

Running the build: Open a terminal session in any shell at the project root — in VS Code use Ctrl+` to open the integrated terminal. The working directory should be C:\Users\napham\OneDrive - Stantec\ProjectBook-Planner. Run the command py tools/build.py and press Enter. Watch the terminal for output. The build script announces each phase as it runs: reading the active site from site-data.json, loading the per-site JSON from data/sites/, reading all site files for __ALL_SITE_DATA__, reading index.json for __SITE_LIST__, inlining CSS, inlining all eight JS engine files, and writing the two output HTML files. When it finishes successfully, the terminal prints a confirmation line and returns to the prompt with no Python traceback. If you see a KeyError, JSONDecodeError, or any Python exception, the build failed. The most common cause is a syntax error in the JSON file you just created — a missing comma after an object property, a trailing comma after the last item in an array, a missing closing brace, or a string with an unescaped double quote inside it. Open the JSON file in VS Code; VS Code highlights JSON syntax errors in the editor and in the Problems panel. Fix the error, save the JSON, and run py tools/build.py again.

Starting the dev server: Run py tools/build.py serve. The script starts a Python HTTP server on port 3040 and prints a message like Serving at http://localhost:3040 — Press Ctrl+C to stop. Leave this terminal open and the server running; do not close it or press Ctrl+C during your session. Open your browser and navigate to http://localhost:3040. The currently active site's InteractiveMap.html will load. Note: port 3034 does not work on this machine because Windows Hyper-V reserves it via HTTP.sys at the kernel level. If you attempt port 3034 you will see a connection refused error. Port 3040 is the correct development URL. The server must stay running for the site-activation POST, the save-to-JSON POST, and the rebuild-after-save behavior to function.

Activating the new site: In the browser, look at the header bar. The site selector dropdown shows the currently active site's label. Open the dropdown — it is a scrollable list populated from window.__SITE_LIST__, which was built from data/sites/index.json in the last build. Your new site should appear in this list by the label you gave it. Click its label. The browser fires a POST /api/sites/{id}/activate request to the Python server, where {id} is the siteId you set. The server receives this, reads site-data.json from disk, replaces the activeSiteId value with the new ID, writes the file back, and immediately runs a full rebuild — reloading the new site's JSON and reinjecting all three globals into both HTML output files. The server returns a 200 OK response. The browser receives the OK and calls location.reload(). The page refreshes completely. On reload, bootstrap.js executes with the new site's data in window.__SITE_DEFAULTS__, ConfigEngine.init() reads the new state, and the Leaflet map opens centered at the lat/lng you set in the JSON.

Verifying the first load: Zoom in to the parcel until you can clearly see lot boundaries and individual buildings in the satellite imagery. If you provided a parcelPolygon array, a yellow boundary polygon should be drawn on the map overlay. If you are using rectangle mode (no polygon), a yellow rectangle will be drawn from the lot width and depth. Verify that the drawn boundary roughly matches the real lot shape visible in the satellite image. The match does not need to be perfect at this stage — you will refine it with the rotation slider next — but the overlay should at least be on approximately the right block. If the overlay appears in completely the wrong location, the lat/lng values you entered may be reversed, or one of them may have had a copy error. Check the JSON, correct the values, rebuild, and reload.

Aligning the lot rotation: Use the rotation slider in the sidebar. Drag it left and right and watch the overlay rotate on the map. Zoom in closely to one corner of the parcel — pick the clearest corner where the lot boundary lines are visible in the satellite imagery. Adjust the slider until the overlay edge at that corner runs parallel to and overlapping the real lot edge in the satellite photo. Then pan to the opposite corner and verify the alignment holds there too. When all four edges (or all polygon edges for irregular lots) are visually aligned, note the rotation angle shown by the slider control. System Detail — How the Engine Works Inside

The mechanics behind chain dimensions, the save pipeline, and LISP export — plus known bugs and their root causes

H1 Chain Dimension System — How Measurements Are Drawn SetbackEngine

Every dimension line you see on the map is drawn by SetbackEngine.updateBldgDimLabels() in engine-setback.js. This function is called after every building drag, after every setback change, and after every redraw. It removes all previously drawn dim layers and redraws the entire system from scratch each time.

The local coordinate system: All geometry is computed in a rotated local frame aligned to the lot. The lot rotation angle (state.rotation in degrees) defines this frame. Two scale factors convert between geographic degrees and feet: F_LAT = 364,566 ft/degree (constant), and F_LNG = 365,228 × cos(lat) ft/degree (varies by latitude). In this local frame, the positive X axis points toward the lot's rear, and the positive Y axis points toward the lot's right side. The lot center is the origin (0, 0) at state.lat, state.lng. Converting a local point (px, py) back to a Leaflet [lat, lng] uses: rx = px × cos(rot) − py × sin(rot), ry = px × sin(rot) + py × cos(rot), then [lat + ry/F_LAT, lng + rx/F_LNG].

The chain concept: Instead of drawing individual per-building dimensions (one line for each building's width, one for its depth), the system draws a single continuous chain across the full lot width and a separate chain across the full lot depth. Each chain is an ordered list of boundary points along one axis. The distance between consecutive points in the list becomes one labeled dimension segment. Shared boundaries between buildings create one witness line that serves both adjacent segments simultaneously — this is what makes it a chain rather than a collection of individual dims.

Collecting boundary points: Two arrays are assembled before drawing begins. dPts holds X positions along the depth axis: it starts with the rear lot edge (−lotHalfDepth) and the front lot edge (+lotHalfDepth), then adds the front and rear edges of every building copy. wPts holds Y positions along the width axis: it starts with the left and right lot edges, then adds the left and right edges of every building. Building edges are cx ± halfDepth and cy ± halfWidth, where halfDepth and halfWidth are the axis-aligned half-extents of the building's bounding box in local space, computed by _buildingExtents(). After collecting all points, both arrays are deduplicated and sorted. The sorted, deduplicated array is the complete chain for that axis.

Perpendicular measurement guarantee: Because building positions and lot boundaries are all expressed in the same rotated local frame, every distance between consecutive chain points is an exact perpendicular measurement — the distance from one face to the next face measured at a right angle to both faces. This is correct regardless of the lot's geographic rotation angle. A distance of 18.0 ft in the depth chain means the gap between those two surfaces, measured perpendicular to the lot's depth axis, is exactly 18.0 feet.

Drawing each chain segment: The inner function drawChain(chain, refCoord, isX, perpX, perpY, rotDeg, prefix) draws one complete chain. refCoord is the perpendicular position at which the dim line runs — for the depth chain, this is an X position offset from the first building's left edge; for the width chain, it is a Y position offset from the first building's front edge. perpX, perpY is the unit vector pointing away from the lot in the direction the dim lines are drawn (either +X, 0 or 0, +Y depending on which side the chain is on). For each adjacent pair of boundary points, the function draws: two short dim line segments split around the text label position, two 45° tick marks at the ends of the segment, and a text label showing the distance in feet. The label is placed at the midpoint of the segment, rotated to align with the axis (rotDeg is −state.rotation so the text stays readable in the browser).

Witness lines: At every boundary position in the chain, a short line is drawn perpendicular to the chain direction, extending from the building/lot edge out to the dim line position plus a 2-foot overshoot (EXT = 5 ft offset plus EX2 = 2 ft overshoot). This witness line is only drawn if at least one of its adjacent segments is visible (not hidden). Witness lines anchor each measurement visually to the exact surface being measured.

Chain repositioning (drag): The user can drag a chain dim line perpendicular to its direction to reposition it for clarity. Dragging updates MapEngine.chainWOffset or MapEngine.chainDOffset in real time. These offsets are snapped to lot edges and building boundaries as the user drags, so the chain dim naturally "docks" at relevant measurement positions. The offsets are saved as part of _payload() and persist across rebuilds.

Click-to-hide: Clicking any dim segment label hides that segment by adding its key to MapEngine.hiddenDimKeys (a Set). The key format is chain_w_3 (width chain, segment index 3) or chain_d_1 (depth chain, segment index 1). On the next redraw, segments with a key in hiddenDimKeys are skipped. The hidden set is persisted in _payload().saved.hiddenDimKeys as an array. On load, it is restored into a new Set() in MapEngine.

H2 Save Architecture — The Single Write Path ExportEngine

The entire application has exactly one save function: ExportEngine.save() in engine-export.js. No other code path writes state anywhere. Every button — Save Setbacks, Save Config, Save Boundary, the FAB button, the Lock toggle — all route to this one function. This guarantees that no matter which action the user takes, the save logic is applied consistently and no path is missed.

Step A — Sync UI inputs to state: Before reading ConfigEngine.state, the function reads every active UI input element and writes its current value back into the state. This handles the case where the user has typed into an input but hasn't pressed Enter — the value would otherwise be in the DOM but not yet in the state object. The function syncs: all four setback inputs, all active building inputs (W, D, offsetX, offsetY, count, stackSpacing, stackAngle, orientation, stories, floorHeight, anchor), the commercial front toggle, and all MapEngine UI state (showBldgDims, hiddenDimKeys, chainWOffset, chainDOffset). After this sync, ConfigEngine.state is the authoritative, complete, current state of the application.

Step B — Build the payload via _payload(): This private method reads ConfigEngine.state and constructs the canonical save object. _payload() is the single source of truth — every field that must survive a page reload must appear here. It is the only place in the codebase that decides what gets saved. The payload wraps everything inside a .saved key along with metadata fields (project, siteId). If a checklist state exists in localStorage under a preapp_* key, it is appended as .checklist — this allows the single save event to capture cross-tool state atomically.

Step C — Write to localStorage: The payload is serialized to JSON and written to localStorage['site_state']. This write happens synchronously and completes in microseconds. It is the offline fallback — if the server is unreachable, or if the user is running in file:// mode without the server, this is the only save path. On the next page load, ConfigEngine.init() reads from localStorage['site_state'] and restores the full state without needing any server communication. The localStorage key is a single flat string — not namespaced by site — so switching to a different site replaces the key entirely. Historical state for other sites in localStorage is not maintained; the JSON file on disk is the permanent record.

Step D — POST to dev server: If the page hostname is localhost or 127.0.0.1, the payload is sent as a POST /save JSON request to the Python dev server at port 3040. The server receives the payload, reads the current site JSON file from data/sites/ using the siteId in the payload, reads the existing .site key from the current file, and writes a new file with the same .site content plus the incoming .saved block overwritten. This is an atomic write: the server writes to a temp file first, then renames it over the original — the existing site JSON is never partially overwritten. After the write, the server immediately rebuilds both HTML output files so the injected __SITE_DEFAULTS__ global in the HTML is current.

Step E — Flash badge: On success, the "Saved" flash badge appears in the top-right corner of the map for 2 seconds. On server error (non-2xx response, or network failure), the badge turns red and shows "Save failed: [reason]" for 4 seconds. The badge resets after the timeout. localStorage was already written in Step C, so even after a server error the state is not lost.

What is NOT saved: The .site key — all the static parcel data filled during the intake process. The live save path never touches it. Changing setbacks, moving buildings, adjusting rotation — none of these touch address, apn, zoning, baseFAR, frontSetback, or any other .site field. Those can only be changed by manually editing the JSON file and running the build.

H3 LISP Export — From GPS Coordinates to AutoCAD Feet ExportEngineMapEngine

The LISP export converts the lot boundary polygon drawn on the Leaflet map into an AutoLISP script that AutoCAD can execute to draw the same boundary at real-world surveying coordinates. The function is ExportEngine.generateLISP() in engine-export.js.

Step 1 — Get the lot polygon vertices: MapEngine.lotPoly is the Leaflet polygon that draws the lot boundary (yellow outline) on the map. Calling MapEngine.lotPoly.getLatLngs()[0] returns the current array of [lat, lng] Leaflet LatLng objects for every corner of the polygon. For a rectangle lot, this is four points. For a polygon site with parcelPolygon, it is however many vertices the polygon has. These coordinates are in WGS84 decimal degrees — the same coordinate system GPS uses.

Step 2 — Register the proj4 projection: proj4.defs(ConfigEngine.cad.projection, ConfigEngine.cad.proj4Def) registers the State Plane projection string into the proj4 library so it can be used in the conversion. ConfigEngine.cad.projection is a human-readable identifier like "CA_VI_FT", and ConfigEngine.cad.proj4Def is the full PROJ.4 definition string for that zone (e.g., for California Zone VI: +proj=lcc +lat_1=32.78... +units=us-ft +no_defs). Both are looked up from ConfigEngine.CAD_SYSTEMS[cadZone] using the cadZone field from the site JSON.

Step 3 — Project each vertex from WGS84 to State Plane: For each lot corner, proj4("WGS84", ConfigEngine.cad.projection, [pt.lng, pt.lat]) takes the GPS decimal-degree coordinates and returns a two-element array of [easting, northing] in US survey feet. Note that proj4 takes the inputs in [longitude, latitude] order (X first, Y second) — not the Leaflet convention of latitude-first. The returned values are real-world State Plane coordinates in survey feet, matching what would appear on a licensed surveyor's boundary map for this parcel.

Step 4 — Format the AutoLISP script: The output is a text string of AutoLISP commands. The header comments embed the APN, rotation angle, and coordinate system name as a reference record. The body is a single (command "_pline" ...) call. Each vertex is written as a quoted "easting,northing" coordinate pair — AutoCAD's pline command accepts coordinates in this format when entered as a string literal. The last argument is "c" (close), which connects the last vertex back to the first to close the polyline ring. Two final lines call _zoom e (zoom extents) and print a success message to the AutoCAD command line.

Step 5 — Deliver the output: The LISP string is placed into the #outputCoords textarea so the user can copy it. A copy of the state (APN, address, lat, lng, rotation, setbacks, LISP text) is written to localStorage['last_lisp_export']. The full save chain (this.save()) fires to persist the current map state. Then a JSON file is downloaded as site-calibration_{APN}.json containing the same state snapshot — this serves as a portable record of what was exported and when.

What the accuracy depends on: The LISP coordinates are only as accurate as the lot polygon visible in the map. For a rectangle site, the polygon is derived from lotWidth × lotDepth and the rotation and lat/lng you set. The accuracy depends entirely on how precisely you aligned the polygon to the aerial: the center pin position determines the absolute coordinates, and the rotation angle determines the orientation. For a polygon site with a parcelPolygon array sourced from the GIS portal, the vertices are from the county's recorded survey, and the alignment depends on how well saved.lat/lng and saved.rotation match the polygon centroid and orientation.

Pasting into AutoCAD: Open AutoCAD. Type APPLOAD, paste the LISP text into a temp .lsp file and load it — or use the AutoCAD command line directly and paste. AutoCAD draws the closed polyline at State Plane coordinates. The polyline's corners should match the parcel corners on any licensed survey for this site within the precision of the alignment step.

H4 Known Bugs — Chain Dims on Slanted and Polygon Lots SetbackEnginebug report

Two active, confirmed bugs affect the chain dimension system for lots that are rotated to non-axis-aligned angles (not 0°, 90°, 180°, 270°) and for lots that use a parcelPolygon array. Both bugs are in SetbackEngine.updateBldgDimLabels(). Cannington works despite being slanted because it is a simple polygon whose visual result happens to be acceptable with the current offset error. El Cajon fails visibly because it is an irregular polygon whose GIS-derived vertices expose the coordinate system mismatch precisely.

Bug KB-1 — Polygon extent query uses the wrong origin (polygon centroid vs. pin position)

The chain dim system needs to know the polygon boundary extent at the X position (or Y position) where the chain line runs. This is done by polyExtentAt(pos, isX), which receives a position and scans across the polygon vertex array to find the min/max crossing point.

The problem is that polyLocalVerts — the polygon vertex array — is computed in a coordinate frame centered on the polygon's geographic centroid (ccLat, ccLng). The centroid is computed inside updateBldgDimLabels() by averaging all polygon vertices. But the position values being queried — chainRefX and chainRefY — are in a coordinate frame centered on state.lat, state.lng, which is the pin the user placed on the map. These two origins can differ by several feet, on any lot where the designer did not place the pin at the exact polygon centroid.

When polyExtentAt(chainRefX, true) is called and the two origins are offset, the function is querying the Y extent of the polygon at an X position that is displaced from where it should be. If the lot is rectangular this mismatch has no visual effect because the polygon spans the full width regardless of where along the depth axis you query. For an irregular polygon — one where the width varies depending on where you are along the depth axis — the returned extent is wrong. The chain endpoints land at the wrong Y positions, causing the chain to visually extend outside the polygon boundary on one side or to fall short on the other.

Root cause code (engine-setback.js, updateBldgDimLabels):

// polyLocalVerts are relative to ccLat/ccLng (polygon centroid)
polyLocalVerts.push({ x: clx, y: cly });   // centroid-relative

// BUT chainRefX is relative to state.lat/state.lng (the pin)
chainRefX = cxFirst - halfDepth;           // pin-relative

// MISMATCH: querying centroid-frame polygon with pin-frame position
var wExt = polyExtentAt(chainRefX, true);  // WRONG when centroid != pin

Fix required: After computing polyLocalVerts, apply an offset that translates from the centroid frame to the pin frame. Compute the centroid's position in local coordinates relative to the pin: dcLat = (ccLat − state.lat) × F_LAT, dcLng = (ccLng − state.lng) × F_LNG, then rotate into local axes and add to every vertex. With this fix, polyLocalVerts[i].x would be in the same coordinate system as chainRefX, and polyExtentAt would return a correct result. Status: not yet fixed. Fix must be validated on Westminster, Cannington, and El Cajon simultaneously.

Bug KB-2 — Witness lines cross the property boundary with no clipping

Every boundary position in a chain (lot edges and building edges) has a witness line drawn from it, perpendicular to the chain direction, extending outward from the lot. The witness line extends EXT + EX2 = 7 ft total from the boundary position in the perpendicular direction. There is no check that this extension stays within the lot polygon.

For the depth chain (which runs along the lot depth axis), witness lines extend in the ± Y direction (the lot width axis). The witness lines at the two lot edges — front and rear — extend outward and appear correctly outside the lot. But the witness lines at intermediate building edges also extend 7 ft in the Y direction from their X position. If a building is at a non-zero Y offset, or if the lot is an irregular polygon where the boundary at that Y offset is not far enough from the building, the witness line will extend past the property line.

For the width chain (which runs along the lot width axis), witness lines extend in the ± X direction. When a building is near the front or rear of the lot, the witness line at the lot's front/rear edge also extends outward, and those extensions appear in front of or behind the lot polygon boundary — visually crossing the property line.

Why this is wrong: The property line is a legal boundary. Any dimension line that crosses it is visually ambiguous — it could suggest a measurement reference point outside the parcel. The rule is that no dimension annotation element should cross a property line. In professional plan drafting, witness lines terminate at or slightly outside the property boundary and never re-enter the lot from the outside.

Fix required: The witness line endpoint must be clamped so it does not cross the nearest property boundary. For rectangle lots, the boundary positions are known exactly (±lotHD, ±lotHW) and the clamp is simple. For polygon lots, the boundary position at the witness line's position on the chain must be looked up via polyExtentAt (once KB-1 is fixed so the frame is correct), and the witness line endpoint trimmed to that boundary value plus a small overshoot. Status: not yet fixed. Depends on KB-1 fix for correct behavior on polygon sites.
