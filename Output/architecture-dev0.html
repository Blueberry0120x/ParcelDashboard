Natural Language Flows

What actually happens — in plain English — for each user journey

FLOW A: FULL ADDRESS INTAKE PROCEDURE
A1 Starting Point: You Have an Address

The designer gives you a street address. That's it — nothing else. No APN, no zoning, no coordinates. Just a string like "2921 El Cajon Blvd, San Diego CA 92104".

Your job is to turn that one string into a fully populated JSON file with 40+ fields. Every field has a specific source. Nothing is guessed. Here's exactly how.

A2 Step 1 — Parcel Identity Lookup

Goal: Find the APN (Assessor Parcel Number), legal description, lot size, and year built.

Open ParcelQuest (assr.parcelquest.com) — this is your primary parcel lookup tool. It's linked in the app's suite bar.

  1. Search by the address the designer gave you.
  2. Verify the result is the correct parcel (check the map pin matches the street view).
  3. Record: APN — this is the unique identifier the county uses. Format varies by state: San Diego uses 471-271-16-00, King County uses 0723049368.
  4. Record: Lot area in SF — ParcelQuest shows this. This is your primary lot size number. Do NOT derive it from width x depth — that only works for perfect rectangles.
  5. Record: Lot dimensions (width x depth) — if shown. These are approximate for rectangular lots. For irregular lots, the polygon geometry is what matters.
  6. Record: Year built — if existing structures are on the lot.
  7. Record: Legal description — the tract, lot, and block reference from the deed (e.g., "LOT 5 BLK 3 MAP 2105").

If ParcelQuest doesn't have it (e.g., Washington state parcels), fall back to:

  • King County: blue.kingcounty.com/assessor
  • Orange County: ocgov.com/gov/assessor
  • San Diego: sdarcc.com (recorder/clerk)
  • Regrid: app.regrid.com — commercial parcel lookup, sometimes has data ParcelQuest misses.
address string Full street address with ZIP. From ParcelQuest or assessor.
apn string Assessor Parcel Number. From county assessor records.
legalDescription string Tract/lot/block from deed. From ParcelQuest "Legal" tab or county recorder.
yearBuilt string Year of existing structure(s). From assessor. Use "" if vacant land.
lotSF number Total lot area in square feet. From assessor — do NOT calculate from W x D.
lotWidth number Front-face width in feet. From assessor or tract map. Use 0 if irregular lot.
lotDepth number Front-to-rear depth in feet. From assessor or tract map. Use 0 if irregular lot.
A3 Step 2 — Coordinates and Parcel Geometry

Goal: Get the exact lat/lng center of the parcel, and — for irregular lots — the full polygon boundary as an ordered list of vertex coordinates.

Getting the centroid (all sites):

  1. Open Google Maps and search the address.
  2. Right-click on the center of the parcel (NOT the building — the lot center).
  3. Click the coordinates at the top of the menu — they copy to clipboard.
  4. Format: decimal degrees WGS84, e.g., 32.755605, -117.091946.
  5. The first number is latitude (saved.lat), the second is longitude (saved.lng).

Getting the parcel polygon (irregular lots only):

If the lot is not a simple rectangle (flag-shaped, L-shaped, angled frontage, multiple combined parcels), you need the actual polygon boundary:

  1. Open the county GIS portal for that jurisdiction:
    • San Diego: SanDag GIS (sandiego.maps.arcgis.com) or SD County GIS
    • Orange County: OC Public Works GIS or Regrid
    • King County (WA): gismaps.kingcounty.gov
  2. Search by APN (found in Step 1).
  3. Identify the parcel boundary polygon on the map.
  4. Extract every vertex as a [lat, lng] pair. Order matters: clockwise from the northwest corner.
  5. Each vertex is a corner of the lot. Each pair of consecutive vertices defines one edge (segment).
  6. Record the length of each segment in feet if the GIS provides it — this helps verify the polygon matches the assessor's lot dimensions.
  7. Store as the parcelPolygon array in the site JSON.

Choosing the CAD coordinate zone:

The CAD zone determines how lat/lng is converted to feet (State Plane projection) when exporting to AutoCAD. Pick based on location:

  • CA_VI — San Diego, Imperial County
  • CA_V — Los Angeles, Orange County, Ventura
  • CA_IV — Fresno, San Luis Obispo
  • CA_III — Sacramento, San Francisco
  • WA_N — Seattle, King County, Snohomish
  • WA_S — Olympia, Pierce, Thurston
saved.lat number Centroid latitude in decimal degrees. From Google Maps right-click.
saved.lng number Centroid longitude in decimal degrees (negative for western hemisphere).
cadZone string State Plane zone code. Based on geographic location, not research.
parcelPolygon array [[lat,lng], ...] Clockwise vertex list from NW corner. Only for irregular lots. Omit for rectangles.
A4 Step 3 — Zoning Code Identification

Goal: Find the zoning designation, then download and read the zoning code to extract the development standards.

Finding the zone code:

  1. Check the county GIS portal — most show a "Zoning" layer. Click the parcel to see the designation (e.g., R-3, CUPD-CU-2-4, RS-1-7).
  2. Alternatively, check the city's zoning inquiry tool:
    • San Diego: ZIMS (sandiego.gov → Development Services → ZIMS)
    • Garden Grove: Municode (codepublishing.com → Garden Grove → Chapter 9)
    • Burien: City code (codepublishing.com/WA/Burien)
    • Escondido: escondido.gov → Planning
  3. Record the exact zone code string. Spelling and hyphens matter — they determine which development standards table applies.

Downloading the zoning code PDF:

  1. Navigate to the city's municipal code for that zone (e.g., San Diego Municipal Code Title 13, Chapter 2).
  2. Find the section that lists development standards for that specific zone — it's usually a table with setbacks, height, FAR, and density.
  3. Download or save the PDF to reference/StateRegulation/{city}/ for future reference.

Extracting development standards from the zoning code:

The zoning code table gives you these specific numbers. Read each one carefully — they drive the entire feasibility analysis:

zoning string Zone designation exactly as stated in municipal code. e.g., "CUPD-CU-2-4", "R-3", "RS-1-7".
baseFAR number Base Floor Area Ratio. How much you can build relative to lot size. lotSF x FAR = max buildable SF. Set 0 for pure residential zones with no FAR limit.
commFAR number Commercial or bonus FAR (e.g., CCHS Tier 3 = 6.5). Set 0 if no commercial component or bonus.
maxHeight number Maximum building height in feet — including any bonus (CCHS, density bonus). This is the ceiling.
baseHeightLimit number Base height WITHOUT bonus. Used as reference; app uses maxHeight for calculations.
cchsMaxHeight number CCHS (Complete Communities Housing Solutions) tier max height. San Diego only. 0 if not applicable.
frontSetback number Minimum front setback in feet. Distance from front property line where you cannot build.
rearSetback number Minimum rear setback in feet.
sideSetback number Minimum side setback in feet. Applied to both sides in the default config.
densityPerSF number Lot SF per dwelling unit. e.g., 600 means 1 unit per 600 SF. Base unit count = lotSF / densityPerSF. Set 0 if density is not SF-based.
commercialDepth number Minimum ground-floor commercial depth in feet (mixed-use zones). 0 for residential-only zones.
A5 Step 4 — Planning Overlays and Special Districts

Goal: Determine which overlay zones and planning designations apply to this parcel. These affect parking requirements, bonus density, height bonuses, and special review triggers.

Each of these is a YES/NO question answered by checking a specific GIS map layer or city document. Check every one — they're stored in the planningAreas[] and overlayZones[] arrays.

Community Plan Area
City general plan or community plan map
Determines which set of land use policies apply (e.g., "Mid-City: City Heights")
Land Use Designation
Same community plan — the specific land use category (Commercial/Mixed Use A, Residential, etc.)
Affects allowed uses and sometimes bonus FAR
CCHS Tier (San Diego only)
sandiego.gov/planning/programs/housing → CCHS interactive map
Tier 1/2/3 unlock bonus height and FAR. Tier 3 = 6.5 FAR + unlimited height in some zones
Transit Priority Area (TPA)
CA: SANDAG GIS transit layers. WA: Sound Transit / PSRC maps.
If YES → may qualify for reduced parking or CEQA streamlining
Parking Standards (PSTPA)
Derived from TPA status + use type. If TPA=Yes and residential → 0 parking required (San Diego).
Directly impacts how many parking stalls you need to build
Mobility Choices
San Diego Mobility Choices zoning map — shows which mobility zone (1, 2, 3)
Higher zone = more transit-oriented parking reductions
Transit Area Overlay
County GIS overlay layer — buffer around transit stops/stations
Additional density bonus eligibility
Airport Influence Area
ALUCP (Airport Land Use Compatibility Plan) map for nearest airport
Height restrictions near flight paths. May limit maxHeight below zoning code allows.
Coastal Overlay
California Coastal Commission maps (documents.coastal.ca.gov) or city coastal zone map
Requires Coastal Development Permit. Adds review layer.
Very High Fire Hazard Zone
CA: Cal Fire FHSZ viewer (osfm.fire.ca.gov). WA: WA DNR fire risk maps.
Triggers fire setbacks, sprinkler requirements, brush management zone
Fault Zone (Alquist-Priolo)
California Geological Survey (maps.conservation.ca.gov/cgs/ap)
Building within fault zone requires geologic investigation. May affect foundation design.
A6 Step 5 — Fee Schedule and Development Costs

Goal: Look up the city's fee schedule to find impact fees, enhancement fees, and affordability requirements.

Every city publishes a fee schedule PDF (usually under Development Services or Permit Center). Search for it:

  • San Diego: sandiego.gov/development-services/forms-publications/fee-schedules
  • Garden Grove: Community Development fee schedule
  • Burien: City of Burien Permit Center fee schedule
  • Washington state: No NEF equivalent — set nefRatePerSF to 0
nefRatePerSF number ($/SF) Neighborhood Enhancement Fee per square foot of new construction. From the fee schedule "NEF" table. Checklist calculates: total NEF = buildable SF x rate. Set 0 if not applicable.
difPerUnit number ($/unit) Development Impact Fee per dwelling unit. From the fee schedule "DIF" line. Checklist calculates: total DIF = qualifying units x rate.
difWaiverSF number (SF) Units at or below this SF are exempt from DIF. From city affordable housing or DIF exemption policy. Set 0 if no waiver.
affordabilityPct number (0-1) Affordability target as decimal. 0.1 = 10% affordable units. From state density bonus law or city inclusionary housing ordinance.
A7 Step 6 — Inspector Contacts

Goal: Get the names and direct phone numbers of the city's plan review inspectors.

Call or email the city's Permit Center. Ask for the assigned inspectors for that area/zone:

  • Combination Inspector — primary building inspector
  • Structural — reviews structural plans
  • Electrical
  • Mechanical — HVAC, plumbing
  • Grading / Civil — site grading, drainage, utilities

If you can't reach them yet, use the TBD format: "-- TBD: City of [X] Permit Center ([phone])" so the designer knows where to call.

inspectors[] array of {name, val} Each entry: name = discipline ("Building Inspector"), val = "First Last (###-###-####)". Pre-populate all 5 rows even if TBD.
A8 Step 7 — Assemble the Skeleton JSON

Goal: Combine all research into a single JSON file following the exact template structure.

File naming:

  1. Filename: {state}-{streetnumber}_{streetname}.json — lowercase state, number as-is, street title-case, no spaces.
  2. Example: ca-2921_ElCajon.json
  3. siteId inside the file: UPPERCASE version: CA-2921_ELCAJON

Copy reference/site-template.json — it has every field pre-set to 0 or "". Fill in what you found:

  • Confirmed data: Use the actual value from your research.
  • Unconfirmed strings: Use "-- TBD: [what it is] | Source: [where to get it]"
  • Unconfirmed numbers: Use 0 — the app handles zero gracefully (shows "--" or "N/A").
  • Arrays (inspectors, planningAreas, overlayZones): Pre-populate all standard rows with TBD values so the structure is ready for the designer to fill.

The notes field is critical — it's the master checklist of what's still TBD:

| TBD - A: legalDescription -- County Recorder or iMap
| TBD - C: commFAR -- verify CCHS tier bonus in zoning code §14.2
| TBD - D: TPA -- check SANDAG transit map layer
| TBD - E: nefRatePerSF -- city fee schedule 2026 edition
| TBD - F: architect, scopeOfWork -- from designer
| TBD - G: all inspectors -- call Permit Center (619) 446-5000
| TBD - H: rotation -- estimate from aerial alignment

Then add to the registry:

  1. Open data/sites/index.json
  2. Add a new entry with: id, label, address, file, state, city, zoning, status: "skeleton", lastUpdated
A9 Step 8 — Build, Activate, and Align

Build: Run py tools/build.py. The pipeline reads every site JSON, injects all 3 globals into the HTML, and writes the compiled output.

Activate: Select the new site from the dropdown (or POST /api/sites/{id}/activate). The server updates the activeSiteId pointer and rebuilds.

Map loads: The satellite view centers on your saved lat/lng. The lot boundary is drawn — either as a rectangle (from lotWidth x lotDepth) or as a polygon (from parcelPolygon).

Align the boundary to the aerial:

  1. Drag the center pin so the lot rectangle/polygon lines up with the aerial photo of the property.
  2. Use the Rotation slider to rotate until the lot edges match the aerial edges.
  3. Set the correct CAD Zone in the COORD dropdown (CA_VI for San Diego, WA_N for Seattle area, etc.).
  4. Click Lock Position when aligned — this prevents accidental moves.
  5. Click Save Boundary — this persists the lat/lng, rotation, and lock state.
A10 Step 9 — Iterate Until Complete

Sites are never done in one session. The designer feeds missing data over days or weeks. Each session:

  1. Check the notes field — it lists every TBD item by checklist section (A–H).
  2. Designer provides new data (e.g., "the zoning is actually CC-3-9, here's the code PDF").
  3. Agent updates the site JSON with confirmed values — replaces TBD strings with real data.
  4. Rebuild. Verify the changes appear in the UI.
  5. Cross off completed items in the notes field.
  6. Update index.json: change status from "skeleton" to "partial" (some TBDs remain) or "complete" (all confirmed).
Status lifecycle:
skeleton Just address + coordinates. Many TBDs.
partial Zoning filled, some sections still TBD.
complete All 8 checklist sections confirmed. Ready for concept design.
